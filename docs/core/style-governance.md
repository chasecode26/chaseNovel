# style：文风治理

`style` 是 `chaseNovel` 第一批优先成形的子 skill 之一。

它负责：
- 文风治理
- 角色说话差分
- 低 AI 味校准
- 风格护栏

它是治理辅链 owner，不是正文生成主链 owner。

## `style` 的边界
- `writer` 负责正文生成
- `style` 负责风格锚点、声口差分、语言风险治理
- `revise` 负责结构、节奏、重复推进、修法判断
- `memory` 负责回写边界与书级状态

先保章节功能，再修语言。

## 总原则
1. 默认现代通俗大白话，先让读者看明白谁在做事、为什么做、结果变了什么。
2. 清晰可读，不等于统一口水化；题材可以带气质，但不能发虚。
3. 对白是行动，不是信息广播。
4. 人物说话必须有差分，不能全员一个作者腔。
5. 先保章节功能，再修语言。

## 硬红线
- 抽象总结句顶替场面
- 作者替角色下结论
- 术语、黑话、判断句不给事实依据
- 故作高深但读者看不懂
- 历史/权谋写成半文半白假古风
- 所有人用同一种成熟顺嘴话
- 用“这才是真刀 / 真正要命的是 / 不是……而是 ……”这类提炼腔替场面做收口
- 问句抛出来后，下一句不正面回答，只拿情绪、气势或身份压人
- 军令已经下了，却不写谁去做、先做什么、慢一步会怎样
- 用“说到底 / 本质上 / 某种意义上 / 这一刻他终于明白”这类总结句替人物体验
- 用“压迫感 / 拉扯 / 博弈 / 张力 / 情绪价值 / 信息差”这类分析黑话替现场信息
- 上一章钉死的时间、人数、伤势、位置、路程、知情边界，下一章直接改口

## 默认检查顺序
1. 先看有没有功能性语言错误
2. 再看信息是否说透
3. 再看角色声口是否撞车
4. 再看是否出现 AI 味 / 省力句 / 提炼句
5. 最后看句子层面的美感和余味

## 连续性硬约束
1. 写下一章前，先复核上一章结尾已经钉死的时间、地点、在场人物、伤势、资源、命令和承诺。
2. 前文说“明天去”，下一章默认只能写“还没到明天”或“已经过夜到明天”，不能直接变成“现在就走”，除非补出中间转折。
3. 前文写“一个打三个人”，后文不能膨胀成“一个打十个人”；要么补援手、地形、兵器、埋伏，要么保持原规模。
4. 人物只能知道自己已经听到、看到、推出来的事，不能突然知道远处还没传到的消息。
5. 命令、伏笔、承诺一旦说出口，后文必须写执行、延后原因或失效原因，不能凭空消失。
6. 新增变化必须有来路：谁带来的、何时发生的、为什么现在才显出来。
7. 战力、人数、时间跨度、路程消耗必须符合场景常识，不能为了强行推进剧情临时改设定。
8. 人和物的位置必须能对上：人物上一章在东门，这一章不能无故出现在中军帐；物资、援军、信使也一样。
9. 任何大动作都要留下后果：夜袭之后会累，翻脸之后会僵，重伤之后会慢，断粮之后会慌，不能下一章像没发生过。
10. 若上一章已经付出代价，这一章必须继续承受、转移或补救，不能直接抹平。

## 最小连续性硬约束集
1. **时间承诺**：上一章已经钉死的今晚、明早、明天、过夜、到达时点，本章若变化，必须补中间过程。
2. **位置与到达**：上一章已经钉死的人物、物资、信使、援手位置，本章若变化，必须补中间过程；来不及到场的人不能硬到场。
3. **后果延续**：上一章已经付出的伤势、疲惫、暴露、翻脸、亏空、断粮等代价，本章必须继续承受、转移或补救；不能直接抹平。
4. **默认回写**：每章写后默认至少回写 `state.md` 与 `timeline.md`；只有涉及伏笔、兑现、人物变化时，才扩展到其他记忆文件。

## 最小 blocking 口径
- 时间承诺对不上，且没有中间原因
- 人物、物资、信使、援手无因换位，或到达时间明显不成立
- 上章后果无因消失
- `state.md / timeline.md` 未回写

## 写后回写原则
- 章节里一旦钉死了新的时间、地点、人数、位置、伤势、资源、消息、命令结果、遗留后果，就必须回写进记忆文件。
- 默认先回写 `state.md` 与 `timeline.md`，不要一开始把所有记忆文件都变成必填负担。
- 不允许把关键连续性信息只留在正文里，靠作者自己记。
- 下一章的稳定性，取决于这一章写后有没有把钉死信息回写干净。

## 低 AI 味默认处理顺序
出现“作者在替场面发话”时，优先按这个顺序改：
1. 用动作替换结论
2. 用异常细节替换气氛词
3. 用代价替换空泛压迫
4. 用人物立场替换作者判词
5. 用结果反馈替换“众人震惊”

常见风险信号：
- 连续几句都在解释感觉，不在写行动
- 旁白比角色更用力
- 高频出现“危险、压迫、冰冷、死寂、震惊、冷笑、瞳孔骤缩”等疲劳词
- 每段都像“先定性，再举例”
- 对话和旁白没有身份差异，谁说都一个味

## 核心载体与 owner 关系
- 单书风格 owner：`templates/core/style.md`
- 书级声音 owner：`templates/core/voice.md`
- 角色说话差分 owner：`templates/core/character-voice-diff.md`
- 风格护栏 owner：`templates/core/style-guardrails.md`

这些模板由 `style` 负责维护口径，`writer` 负责消费，`revise` 可在修章时引用，`memory` 不负责改写这些风格规则。
