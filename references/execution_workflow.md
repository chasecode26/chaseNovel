# 核心执行工作流与门禁机制

这份文档定义写每一章前、中、后必须执行的动作，以及默认门禁规则。

目标不是让流程变重，而是让章节在进入正文前就锁住功能、连续性、文风和结果变化。

---

## 1. 写前准备

开始写任何章节前，必须先完成以下锚定：

1. 读取 `plan.md` 与 `state.md`
2. 明确 `time_anchor`
3. 明确 `location_anchor`
4. 明确 `present_characters`
5. 明确 `scene_focal_character`
6. 明确 `knowledge_boundary`
7. 明确 `resource_state`
8. 明确 `open_threads`
9. 明确 `forbidden_inventions`
10. 明确本章功能、结果变化、章尾钩子、字数等级

若其中任一关键项答不清，先补 memory 文件，不进正文。

## 1.1 章节规划预审

`Writer` 起草前，必须先过一轮规划预审。

以下任一项答不清，默认 `revise`：

- 本章为什么必须存在
- 本章功能是否唯一
- 本章是否与近 `3-5` 章重复同一种推进
- 本章是否有明确结果变化
- 本章钩子是否会改变下章行动
- 关键转折是否已有前提
- 若涉及危险、身份暴露、伏笔触发，既有后手是否会被调动
- 本章是否在用黑话、简称、半截判断替代可判读信息

预审结论只有两种：

- `pass`
- `revise`

未过预审，不准进正文。

## 2. 多 Agent 生产顺序

长篇连载默认使用以下顺序：

`Planner -> HookEmotion -> Writer -> (LanguageReviewer || StyleDialogue || ContinuityReviewer || CausalityReviewer) -> Reviewer`

这里的 `Reviewer` 指总审角色，负责最终放行，不等于并行 reviewers。

补充规则：

- 同一章同一时刻只允许一个 `Writer`
- reviewer 并行，只审不抢写
- `Reviewer` 拥有最终放行权
- 任一 reviewer 给出 `blocking=yes`，直接回退，不准继续下章

## 3. 起草规则

### 3.0 Writer 自检

交稿前，`Writer` 先问自己：

- 有没有作者提炼句、概括句、讲道理句顶掉场面
- 有没有把所有角色写成同一种口气
- 有没有把关键信息写成黑话、简称、半截推理
- 有没有称谓、身份、官职、场合打架
- 有没有为了赶进度先写省力句，指望后面再修

只要任一项为“有”，先返工，不交复审。

### 3.1 正文底线

- 先场景，后解释
- 先行动，后信息
- 先冲突，后设定
- 判断句必须先有事实，再有判断，再落后果
- 对白必须让读者看出谁在压、谁在让、谁在换条件
- 每章至少有一个明确的结果变化
- 章尾必须留下新压力或可执行钩子

### 3.2 反 AI 味硬规则

- 禁用提炼句硬顶场面：
  - `真正的问题是`
  - `关键在于`
  - `本质上`
  - `说到底`
  - `这意味着`
  - `这说明`
  - `说白了`
- 禁用机械书面语：
  - `由于`
  - `导致`
  - `不禁`
  - `不由得`
  - `突然之间`
  - `值得一提的是`
- 禁把黑话和简称当读者常识
- 像“刀”这类简称，若承担关键信息，必须同句或下一句翻成人话
- 禁故作高深式留白去遮普通信息

### 3.3 场景与文风平衡

- 场景最低配必须含 `动作 + 依据 + 后果`
- 开头第一屏必须有事，主角尽快入场
- 强情节优先短句，氛围烘托才允许长句
- 细节不足时补环境、身体反馈、资源变化、动作细节
- 氛围过载时删抽象铺陈，保留推进动作和结果变化
- 留白只给悬疑必要处，不给普通信息

## 4. 写后同步

章节通过复审后，按需同步：

1. `summaries/recent.md`
2. `timeline.md`
3. `state.md`
4. `foreshadowing.md`
5. `arc_progress.md`
6. `character_arcs.md`
7. `payoff_board.md`

若章节未通过，不做正式回写。

## 5. 门禁检查

脚本、黑名单、审卡都只是辅助证据。
最终是否过关，以 `LanguageReviewer` 与 `Reviewer` 的人眼结论为准。

### 5.1 Reviewer 通篇复核纪律

- 必须先整章顺读
- 再按六类拆检：
  - 逻辑不通
  - 表达不严谨
  - 词语重复
  - 说话者语气不对
  - 作者视角发声
  - 术语省略
- 任何一类命中，默认 `verdict: revise`
- 复核结论必须写清：
  - 退回给谁
  - 先修哪一层
  - 修完先复查什么

### 5.2 必过项

- 不违背 `plan.md`
- 时间线可落到 `timeline.md`
- 人物行为与性格相符
- 本章有结果变化
- 开头第一屏有事
- 章尾钩子具体
- 高点前有蓄压，高点后有余波或新压力
- 没有明显拖段、讲段、重复推进
- 没有明显 AI 味、假氛围、黑话挡路、半截推理
- 人眼读感像人写，不像作者在台前提炼

### 5.3 阻断项

命中任一项，直接阻断：

- 节奏崩塌
- 时间线冲突
- 人设失真
- 既有后手、布置、伏笔应触发却完全没动
- 本章该兑现却写成一整章解释
- 连续多章同质推进
- 整段故作高深
- 对白发虚
- 叙述发虚
- 术语挡路
- 半截推理
- 逻辑断裂
- 口气错人
- 作者出声
- 水章节
- 写前规划未过审

## 6. 战场 / 权谋双闸

若章节涉及军报、敌情、权谋、推理，`Writer` 交稿后，`Reviewer` 前默认加双闸：

### `language_block`

检查：

- 黑话是否顶关键信息
- 首现术语是否就地解释
- 句子是否一遍能懂
- 问答是否答题
- 命令是否能执行

输出：

- `plain_language_pass: yes | no`
- `term_explained_on_first_use: yes | no`
- `qa_matched: yes | no`
- `order_can_execute: yes | no`

### `causality_block`

检查：

- 判断前是否先给事实
- 判断后是否交代后果
- 主角推理依据是否落地
- 是否同时考虑敌我条件
- 是否存在需要读者自己补完的逻辑断口

输出：

- `fact_judgment_consequence_clear: yes | no`
- `protagonist_reasoning_clear: yes | no`
- `shared_conditions_checked: yes | no`
- `reader_inference_gap: yes | no`

裁决：

- 任一项为 `no`，默认 `verdict: revise`
- 双闸都过，也不等于自动放行，仍需 `Reviewer` 通读裁决
