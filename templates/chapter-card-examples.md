# 完整章卡示例

> 用途：给 `/写` 提供可直接套用的完整章卡参考。
> 原则：章卡不是摘要，而是正文开写前的执行卡。要写清功能、冲突、结果变化、角色推进、关系刷新、承诺推进与章尾钩子。

---

## 1. 玄幻 / 修仙：升级反打章

```markdown
# 第 12 章写作方案

## 1. 本章章卡
- 本章功能：兑现小爽点 + 推进主线
- 本章目标：让主角拿回本该属于自己的入山资格
- 本章冲突：外门执事临时改规，想把名额转给内定弟子
- 本章结果变化：主角不但保住资格，还让执事当众丢脸
- 本章爽点 / 情绪点：被压制后的第一次正面反打
- 本章主要推进角色：主角
- 本章关系刷新点：围观弟子对主角态度，从轻视转为警惕
- 本章承诺推进 / 延后说明：推进“主角会在宗门翻身”的核心承诺
- 本章章尾钩子：真正负责考核的长老突然点名，让主角单独留下

## 2. 正文草稿
- 开头先给现场压制：名额将被拿走
- 中段让主角拿出前文埋下的证据或规则漏洞
- 当众反打，必须让局面发生可见变化
- 结尾不要停在赢，而要接更高一级的压力

## 3. 门禁结论
- 是否有明确推进：是
- 是否有结果变化：是
- 是否推进角色或关系：是
- 是否符合番茄向快反馈要求：是
- 是否存在水章风险：低
- 是否存在重复风险：低

## 4. 需更新的记忆点
- `state.md`：主角已拿到入山资格，下一章进入考核线
- `recent.md`：记录本章反打结果与长老点名
- `character_arcs.md`：主角首次建立“可被正视”的位置变化
- `payoff_board.md`：宗门翻身承诺已部分兑现
```

## 2. 都市 / 系统流：任务打脸章

```markdown
# 第 8 章写作方案

## 1. 本章章卡
- 本章功能：兑现爽点 + 强化系统价值
- 本章目标：让主角在会议上拿下原本不可能拿到的客户
- 本章冲突：上司故意甩锅，同事等着看笑话，客户极难搞
- 本章结果变化：主角完成任务，客户改签给他，上司失去话语权
- 本章爽点 / 情绪点：系统介入现实，立刻改写主角处境
- 本章主要推进角色：主角
- 本章关系刷新点：女同事从看热闹变成开始主动接近
- 本章承诺推进 / 延后说明：推进“系统会帮主角翻身”的承诺
- 本章章尾钩子：系统刷新出一个更危险但回报更高的连锁任务

## 2. 正文草稿
- 开头先落任务和羞辱，不先解释系统新功能
- 中段让主角靠系统给出的关键提示完成破局
- 爽点必须落在客户、会议室、上司表情这些现实反馈上
- 章尾给更高回报，也给更大风险

## 3. 门禁结论
- 是否有明确推进：是
- 是否有结果变化：是
- 是否推进角色或关系：是
- 是否符合番茄向快反馈要求：是
- 是否存在水章风险：低
- 是否存在重复风险：注意避免只写系统播报

## 4. 需更新的记忆点
- `state.md`：主角在公司地位上升，下一章进入连锁任务
- `recent.md`：记录会议翻盘与客户转签
- `characters.md`：更新上司、同事对主角的新态度
- `payoff_board.md`：翻身承诺推进，资源线已打开
```

## 3. 甜宠 / 总裁：关系升温章

```markdown
# 第 15 章写作方案

## 1. 本章章卡
- 本章功能：推进关系线
- 本章目标：让男女主关系从“互相试探”推进到“公开偏爱”
- 本章冲突：女主被当众针对，男主若出手会暴露立场
- 本章结果变化：男主当众护短，关系第一次半公开
- 本章爽点 / 情绪点：偏爱落地，第三方见证
- 本章主要推进角色：男主、女主
- 本章关系刷新点：两人从暧昧试探进入明确站队
- 本章承诺推进 / 延后说明：推进“男主偏爱会越来越明显”的感情承诺
- 本章章尾钩子：女主回去后发现，男主早就替她处理掉了更大的麻烦

## 2. 正文草稿
- 开头先落社交场或公开场合的压力
- 中段让针对升级，逼男主必须做选择
- 关键动作必须具体，比如挡酒、牵手、改口、公开称呼
- 结尾补一个更私人的后续动作，形成追读

## 3. 门禁结论
- 是否有明确推进：是
- 是否有结果变化：是
- 是否推进角色或关系：是
- 是否符合番茄向快反馈要求：是
- 是否存在水章风险：低
- 是否存在重复风险：注意不要只写心理心动

## 4. 需更新的记忆点
- `state.md`：关系进入半公开阶段
- `recent.md`：记录公开护短事件
- `character_arcs.md`：关系弧推进一格
- `payoff_board.md`：感情承诺获得小兑现
```

## 4. 悬疑 / 刑侦 / 推理：线索翻案章

```markdown
# 第 21 章写作方案

## 1. 本章章卡
- 本章功能：推进主线 + 刷新认知
- 本章目标：推翻上一章形成的嫌疑人判断
- 本章冲突：现有证据几乎全部指向错误对象，主角团队内部意见分裂
- 本章结果变化：一条时间线证据被翻出来，原嫌疑方向被推翻
- 本章爽点 / 情绪点：认知翻盘，读者原有判断失效
- 本章主要推进角色：主角、关键搭档
- 本章关系刷新点：搭档第一次承认主角的判断是对的
- 本章承诺推进 / 延后说明：推进“真相远比表面复杂”的悬念承诺
- 本章章尾钩子：新证据最后指向一名已经‘死了’的人

## 2. 正文草稿
- 开头先重申当前错误共识，让读者进入旧判断
- 中段通过一个细节证据制造时间矛盾
- 不要靠口头推理撑全章，要有证据落地
- 结尾必须抛出新的更大谜面

## 3. 门禁结论
- 是否有明确推进：是
- 是否有结果变化：是
- 是否推进角色或关系：是
- 是否符合番茄向快反馈要求：是
- 是否存在水章风险：低
- 是否存在重复风险：注意不要连续多章都只做信息推翻

## 4. 需更新的记忆点
- `state.md`：案件判断方向改变
- `recent.md`：记录翻案证据与新嫌疑线
- `timeline.md`：更新时间线矛盾点
- `foreshadowing.md`：记录“已死之人”新伏笔
- `payoff_board.md`：真相承诺继续升压
```

## 使用建议

- `/写` 前如果不知道章卡该怎么落，可以先选最接近题材的示例改写
- 先照结构写，不要先追求文气
- 章卡至少要保证“功能、冲突、结果变化、钩子”四项不缺
- 如果章节偏关系线或偏悬疑线，再额外强化“关系刷新点”或“认知变化点”
